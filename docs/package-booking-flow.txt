### Validation Rules (Screen PB-1)
-----------------------------------------------------------
- Package type: **Required**
- Accommodation type: **Required**
- People count: **1 – 40**
- Customer name: **Required**, letters + spaces only
- Email: **Required**, valid email format
- Phone: **Required**, 10-digit Indian mobile (starts with 6-9)
- Each person:
    • Name (required)
    • Age (5 – 80)
    • Activity (required – Surf or SUP)
- Activity capacity limits:
    • Surf ≤ 40 per slot  
    • SUP ≤ 12 per slot  
  → If exceeded, show warning and block submission

-----------------------------------------------------------
### API Calls (Screen PB-1)
-----------------------------------------------------------
**On Load**
→ `GET /utils/package-info`  
Fetch available package types, durations, and base pricing for Tent, Dorm and Cottage options.

**On Continue**
→ Client-side validation only (no API yet)  
If all inputs valid and within capacity → proceed to Screen PB-2 (Session & Slot Selection).

-----------------------------------------------------------
### Navigation (Screen PB-1)
-----------------------------------------------------------
← **Back to Booking Options** → Return to main booking page  
→ **Continue to Select Sessions →** → Validate → Load Screen PB-2

### Validation Rules (Screen PB-2)
-----------------------------------------------------------
- **Check-in date:** Must be within the defined weekly booking window  
- **Sessions:** Each session must have a valid slot for every activity  
- **Capacity per session:**  
  • Each activity must have enough available spots for the assigned people  
  • Validate dynamically whenever activities are customized  

- **Session Dates:** Automatically determined by selected package type:  
  • 1N1S → 1 session (on checkout day)  
  • 1N2S → 2 sessions (check-in + checkout days)  
  • 2N3S → 3 sessions (check-in day + next 2 days)

-----------------------------------------------------------
### API Calls (Screen PB-2)
-----------------------------------------------------------

**On Load:**  
→ `POST /utils/generate-sessions`  
**Payload:**
```json
{
  "package_type": "1N2S",
  "check_in_date": "2025-10-23",
  "people_count": 5,
  "people": [
    {"name": "John Doe", "activity_type": "surf"},
    {"name": "Jane Doe", "activity_type": "surf"},
    {"name": "Bob Smith", "activity_type": "surf"},
    {"name": "Alice Johnson", "activity_type": "sup"},
    {"name": "Mike Davis", "activity_type": "sup"}
  ]
}
Returns:
Sessions array with pre-assigned slots for each activity.

When user changes Check-in Date:
→ Call POST /utils/generate-sessions again
→ Regenerate sessions and reassign default slots based on new dates.

When user customizes activities:
→ Validate locally for capacity per activity
→ If capacity OK, apply local change
→ Trigger re-fetching and reassignment of slots for affected session(s):
• System will call POST /utils/generate-sessions internally
• New available slots will be automatically reassigned based on updated composition
• User can modify further using “Change Slot” if desired.

When user changes slot:
→ GET /slots/activities?date={date}&activity={type}&people={count}
Fetch available slots for selected activity on chosen session date.

Before Continue:
→ POST /utils/validate-package-capacity
Payload: Full sessions array with all assigned slots and activities
Validates:
• Each activity has sufficient capacity in its assigned slot
• Returns error if any slot is overbooked or unavailable.
